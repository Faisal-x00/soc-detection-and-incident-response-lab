# Windows Threat Hunting Notes

This document covers findings from reviewing Windows endpoint activity in this lab beyond the five planned detection scenarios, using Wazuh's archives index to review all processed events rather than only those that generated alerts.

## Finding 001: False Positive, Disk Cleanup Temp File Drop

**Rule Triggered:** 92213, Executable file dropped in folder commonly used by malware
**Rule Level:** 15, Critical
**Alert Volume:** Initially 26 hits, later observed up to 56 hits in a short burst

### What Happened
A burst of critical severity alerts fired from rule 92213, which flags executables or DLLs written into folders commonly abused by malware for staging payloads, such as paths under AppData\Local\Temp.

### Investigation
The triggering process was identified as `C:\Windows\System32\cleanmgr.exe`, the native, digitally signed Windows Disk Cleanup utility. The file it dropped was `WimProvider.dll`, a legitimate Windows Imaging component extracted as part of normal Disk Cleanup operation, written to a randomly named subfolder under the user's Temp directory.

### Verdict
False positive. The rule's logic is sound in general, since Temp folder executable drops are a legitimate high-value indicator for malware staging, but in this specific case the actor was trusted, native Windows system maintenance activity rather than an attacker.

### Tuning Recommendation
Consider an allowlist exception in rule 92213 for known legitimate parent processes such as cleanmgr.exe, TiWorker.exe, and other Windows Update or system maintenance binaries, to reduce alert noise without weakening detection of drops from unknown or unsigned processes.

## Finding 002: Windows Update Background Noise

**Rules Triggered:** 92217, Executable dropped in Windows root folder, and 92219, Possible DLL search order hijack by files created in Windows root folder
**Rule Level:** 6 for both
**Alert Volume:** Large burst, roughly 85 pages of results at the time of review

### What Happened
A large burst of alerts fired referencing files such as fvereseal.dll, WinREAgent.dll, ReserveManager.dll, TurboStack.dll, wcp.dll, and UAOneSettings.dll, all created under a path resembling `C:\Windows\SoftwareDistribution\Download\<GUID>\Metadata\`.

### Investigation
`SoftwareDistribution\Download` is the standard staging folder Windows Update uses to download and extract update packages prior to installation. The burst coincided with the lab endpoint catching up on a backlog of pending updates after being powered off for an extended period, which is consistent with normal, automated Windows Update behavior rather than attacker activity.

### Verdict
False positive, and already reflected in the rule's own low severity level of 6, which Wazuh treats as informational or low priority relative to the level 12 to 15 alerts generated by the actual simulated scenarios in this project.

### Tuning Recommendation
No rule change is strictly necessary given the already-low severity, but an analyst reviewing a large burst from these rule IDs should check the file path for the SoftwareDistribution\Download pattern early in triage, since it is a fast, reliable way to rule out the majority of Windows Update-related noise without needing to inspect every individual file.

## General Note on Method
Both findings above were identified using Wazuh's Threat Hunting interface filtered to the windows-endpoint agent, sorted by rule level, and cross-referenced against the exact file and process fields in each alert rather than relying on the rule description alone. This same triage process, checking the actual triggering process and file path before accepting or dismissing an alert, was applied consistently across both these organically-occurring findings and the five planned detection scenarios in this lab.
