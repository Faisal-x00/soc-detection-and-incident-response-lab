# Final Project Report

## Overview
This report summarizes the SOC Detection Engineering and Incident Response Lab in full, consolidating the methodology, environment, per-scenario outcomes, and cross-cutting findings documented throughout this repository. It is intended as a single, complete technical record of the project, complementary to the shorter executive summary and the individual scenario documentation.

## Objective
The objective of this project was to build a portfolio-quality demonstration of the complete SOC analyst and detection engineer workflow, covering detection gap analysis, custom rule development, alert investigation, timeline reconstruction, and incident reporting, rather than a narrower demonstration of SIEM configuration alone.

## Environment
The lab was built entirely within an isolated VMware environment on a single host machine, using host-only and NAT networking with no exposure to public networks. Three virtual machines were used: a Wazuh manager running an all-in-one install of the Wazuh manager, indexer, and dashboard on Ubuntu; a Windows 10 endpoint instrumented with Sysmon using the SwiftOnSecurity configuration, alongside a Wazuh agent forwarding both Sysmon and native Windows Event Log channels; and a second Ubuntu endpoint running a Wazuh agent monitoring native SSH authentication logs. Wazuh's archive feature was enabled partway through the project to provide full visibility into all processed events, not only those that generated alerts, which proved necessary for debugging custom rule behavior and useful for subsequent threat hunting.

## Methodology
Each of the five scenarios in this project followed the same structure. First, a specific attacker behavior was simulated safely, using non-destructive methods and, where applicable, harmless stand-in payloads such as simple echo commands or the legitimate Windows Calculator binary rather than any real malicious code. Second, the resulting telemetry was reviewed to determine what, if anything, Wazuh's built in ruleset already detected. Third, the specific logic of any relevant built in rule was examined directly, rather than assumed, to identify precisely what gap, if any, existed in its coverage. Fourth, where a genuine gap was found, a custom rule was designed, tested, and iterated until verified working against live data. Fifth, each scenario was documented with raw log evidence, an investigation narrative, a timeline, recommended response actions, and a short incident report summary.

## Scenario Summaries

**Scenario 001, PowerShell Encoded Command.** Wazuh's built in rule 92057 detects base64-encoded PowerShell execution only when the parent process is also PowerShell. Custom rule 100002 extends detection to a defined set of non-PowerShell parent processes commonly abused to launch encoded commands, closing the gap. Development required correcting an initial approach based on field negation, which loaded without error but did not match live events.

**Scenario 002, New Local Administrator Account Creation.** Wazuh's built in rules detect account creation and Administrators group changes as separate, individually low-signal events. Custom rule 100010 correlates the two within a two minute window, escalating severity for the specific sequence of account creation immediately followed by privilege escalation, which is a stronger indicator than either event alone. This rule worked correctly on the first implementation attempt.

**Scenario 003, Suspicious Windows Service Creation.** Wazuh's built in rule 61138 detects any new service creation at a flat, low severity regardless of what the service executes. Custom rule 100020 chains off this rule and escalates severity specifically when the service's executable path references a known interpreter or living-off-the-land binary. This rule also worked correctly on the first attempt, and the same underlying event was independently confirmed by a separate File Integrity Monitoring rule watching the relevant registry key, demonstrating layered detection coverage.

**Scenario 004, Startup Folder Persistence.** Wazuh's built in rule 92204 detects PowerShell-created executables broadly under a user's AppData directory, but does not specifically distinguish the Startup folder, a considerably higher-confidence persistence location, and only fires when PowerShell specifically was the creating process. Three separate, valid custom rule designs were built and tested to escalate Startup-folder-specific detections, all of which loaded successfully but none of which took precedence over or coexisted with the existing alert in the live pipeline. This outcome is documented as an identified gap with a well-reasoned but unsuccessful fix attempt, attributed to an underlying limitation in Wazuh's rule correlation engine rather than a flaw in the rule logic itself.

**Scenario 005, SSH Brute Force.** Wazuh's built in ruleset already provides strong, layered coverage of SSH brute force activity, including a dedicated rule for failures followed by a success. A more subtle gap was identified in that this correlation rule has no enforced frequency threshold, meaning it technically fires after only one failure and a success. Custom rule 100040 was built to independently flag high volume failure activity regardless of whether a success ever follows, chaining off a different rule than the one used by the built in correlation rule after an initial same-chain attempt did not fire, consistent with the same limitation observed in Scenario 004.

## Cross-Cutting Findings
Two findings recurred across multiple scenarios rather than being isolated to one. First, several built in Wazuh rules scope their detection logic more narrowly than the underlying attacker behavior warrants, whether through a specific parent process requirement, a broad file path match that does not distinguish higher-risk subpaths, or a description implying a threshold that is not actually enforced in the rule's logic. Second, custom rules built to share an existing correlation chain with a built in rule, using if_group and if_matched_group together, were observed in two separate scenarios to fail to take precedence over or coexist with the built in rule's alert, despite loading and validating without any configuration error.

Beyond the five planned scenarios, two real, organically-occurring false positives were investigated using the same triage discipline applied throughout this project, both involving legitimate Windows system maintenance activity, cleanmgr.exe and Windows Update, being flagged by rules designed to catch generically suspicious file drop locations. Both are documented in the Windows threat hunting notes alongside tuning recommendations.

## Conclusion
This project demonstrates that meaningful detection engineering value can be delivered not only by writing new detection rules, but by rigorously evaluating existing coverage first, and that recognizing a tool's genuine limitations and documenting them clearly is as legitimate an output of this work as a fully working rule. Four of five scenarios resulted in verified, working custom detections closing real gaps in default coverage. The fifth resulted in a well-documented, honestly reported limitation. Recommendations for each finding are detailed in the corresponding scenario documentation and consolidated in the detection coverage matrix.
