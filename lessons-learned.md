# Lessons Learned

## On Gap Analysis as a Methodology
Approaching each scenario by first identifying what Wazuh's built in ruleset already covers, rather than writing detection rules from scratch, proved to be the most valuable structural decision in this project. It surfaced narrow scoping decisions in the built in ruleset that would not have been visible if custom rules had simply been written in isolation to detect each simulated behavior. It also produced one legitimate case, Scenario 5, where the honest finding was that existing coverage was already strong, which is itself a useful and realistic outcome for a detection engineer to be able to recognize and document rather than always defaulting to building something new.

## On Rule Correlation Behavior in Wazuh
Two separate scenarios, 004 and 005, encountered the same underlying issue: a custom rule sharing an existing correlation chain, via if_group and if_matched_group, with a built in rule did not fire as expected, even though the rule loaded without any configuration error and validated cleanly through wazuh-analysisd. In Scenario 5, this was resolved by chaining the custom rule off a different, unrelated rule instead. In Scenario 4, three different valid rule designs were attempted, and none resolved the issue, resulting in an honestly documented limitation rather than a working fix. This appears consistent with a known category of inconsistency reported by other Wazuh users in similar rule correlation scenarios, though the exact root cause was not conclusively identified within the scope of this project.

## On Negation Logic
Scenario 1's first custom rule attempt used a negated field match, `negate="yes"`, intended to fire on any parent process other than PowerShell. This loaded without error but never matched live events. Switching to an explicit allowlist of known non-PowerShell parent processes resolved the issue immediately. The exact cause of the negation approach's failure was not fully resolved, but the practical lesson is to test negated conditions explicitly against real captured events before relying on them, rather than assuming standard regex negation semantics carry over cleanly into Wazuh's field matching engine.

## On Character Transcription Errors
Several debugging sessions in this project were ultimately caused not by logic errors but by single-character transcription mistakes made when manually retyping rule XML into a terminal editor, including a case where `(?i)`, the standard case-insensitive regex flag, was mistyped as `(?1)`, a valid but entirely different PCRE2 construct that compiled without error but broke the intended match. The practical lesson is that copy-pasting verified content directly, rather than retyping it by hand, meaningfully reduces this class of error, and that a rule loading without a configuration error is not sufficient confirmation that the rule's logic is correct.

## On Severity Calibration
Rather than assigning arbitrary severity levels to custom rules, this project attempted to reason about severity relative to neighboring existing rules. Rule 100040 in Scenario 5 was deliberately set to level 11, between the existing rule 5551 at level 10 and rule 40112 at level 12, reflecting that high volume login failures without a confirmed success are more significant than generic frequency alone but less severe than a confirmed successful compromise. This kind of relative reasoning is treated here as a more defensible practice than copying a nearby rule's severity value without justification.

## On the Value of Archives and Full Event Visibility
Early debugging in Scenario 1 was significantly slowed by the fact that Wazuh's default alerts index only stores events that already matched a rule, meaning a custom rule that fails to match produces no visible record at all. Enabling Wazuh's archive feature, which logs all processed events regardless of match status, was necessary to confirm that events were reaching the manager correctly and to directly observe what fields were present, and it remained useful afterward for general threat hunting, including both false positive findings documented in the threat hunting notes.

## On Recognizing and Documenting Tool Limitations
Scenario 4 did not end in a working custom rule. Rather than treating this as an incomplete or failed scenario, it is documented here as a deliberate choice: recognizing a genuine limitation in a tool, attempting a reasonable fix through multiple valid approaches, and clearly documenting both the limitation and the reasoning behind the attempted fixes is treated as a legitimate and realistic outcome of real detection engineering work, not a gap in the project itself.
